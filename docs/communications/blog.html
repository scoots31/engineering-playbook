Solo Builder Framework

Release Notes

What changed in the framework, what was added, and why — in plain language.

April 25
Delivery Rules — Commands and Review
New
AI runs terminal commands — never the solo
The framework never pastes a command for the solo to copy and run. If a command needs to execute, the AI runs it. If the action is significant or destructive, the AI asks permission first — then runs it. Applied as a load-bearing rule in the Output Contract, framework principles, solo-build skill, and both installed Cursor rules.
New
Review is always visual
When a build produces UI or HTML output and review is needed, the AI serves it as a viewable page before asking for feedback. The solo looks at the work, then responds. Asking for sign-off on a description of what was built is not review. Same six-file cascade as the terminal commands rule.
New
Release notes blog launched
This page. Framework improvements are now tracked in plain language — what changed, what was added, and why — rather than living only in git commit messages.
New
Deliverable — new framework concept
The framework now has a named layer between the plan and individual slices. A deliverable is the body of work a set of slices collectively produce. Two types: screen (reviewed visually in the browser) and logic (reviewed against evidence — test results, data state, or a simulated flow through affected screens). Three fields per deliverable: technical spec for the AI, solo description for the human, and acceptance criteria as the shared contract both parties build and review against. Documented across the framework doc, QA guide, Phase Test guide, skills reference, and getting-started.
New
Two review levels
Slice approval (did this unit of work complete correctly) and deliverable acceptance (is the full body of work done and does it match the agreed contract) are now distinct moments. The AI self-verifies against acceptance criteria before presenting a review to the solo. QA runs a deliverable-level check once all slices in a deliverable reach Done — screen deliverables get a visual pass, logic deliverables get an evidence pass. Phase Test walks each deliverable in the phase and confirms they hold together in integration.
New
"show progress" and "show plan"
Two plain-language phrases available at any time during build. "show progress" returns the active deliverable — all slices, current status, acceptance criteria. "show plan" returns the full build — all deliverables, delivered / in progress / backlog. Surfaced automatically at key moments; also available on demand. Noted in the getting-started Build phase row so solos know the phrases from day one.
Improved
Session modes renamed: bare / piloted / guided
The three session modes were renamed from bare / assisted / auto-pilot. "Auto-pilot" created the wrong mental model — people assumed the AI worked independently while they stepped away. The new names reflect what each mode actually means: bare (you invoke, nothing pre-loads), piloted (you're at the controls, always-on instruments are live), guided (the framework leads the phase sequencing, you're in every decision). Cascaded across 18 files — framework doc, all always-on skill files, template, setup prompts, and every communications page.
New
Dedicated modes slide in the solo deck
New Slide 5 explains bare / piloted / guided with one paragraph per mode — clear enough that the mental model lands before the always-on slide references them. Includes an enhanced mode callout at the bottom explaining it as a separate capability layer (Claude Code only, multi-agent orchestration) that can combine with any session mode.
New
Data integration planning — data questions log + integration deliverables
The mock→real data swap is no longer treated as a checklist item at phase test. It's now explicitly planned from design review onward. Design review gets a named data behavior pass — required for every screen with external data — that asks scale and behavior questions upfront: will this list paginate, which fields can be null, what does the empty state look like? Every unanswered question goes into a data questions log that gates dependent UI slices from reaching Ready. Mock data is updated after questions resolve, so UI slices are built against realistic conditions from the start. In planning, every Screen deliverable with external data gets a companion Logic deliverable for data integration — always a separate deliverable, always sequencing after the Screen is accepted, scoped only once the data questions log is resolved. Standard done criteria for integration slices cover API shape, data-mapping.md confirmed, pagination and filtering, empty and error states, loading states, and auth. At integration deliverable acceptance, solo-qa runs a UX impact check: does live data produce anything a user would see that the design didn't account for? If yes, the affected UI slices reopen and design review runs before the deliverable is accepted. Early questions, separate deliverables, late confirmation.
New
process-change — to-be map change management
New skill providing a consistent protocol for changes to the agreed to-be process map, regardless of cause or where in the build they surface. Stakeholder input, demo gaps, technical constraints, and solo realizations all receive the same seven-step response: capture trigger → document change → three-tier impact scan (scheduled, in-progress, and completed work — nothing assumed safe) → solo decides → execute → re-prioritize → solo confirms before resuming. Wired into solo-build (map-level conflict auto-detection), qa-triage (new map-level gap classification), and phase-test Stage 6 (process-level mismatch blocks the gate). The process is the contract. When it changes, it changes deliberately.
April 24
Output Discipline & Design Sprint Precision
New
Output Contract — two new load-bearing principles
Explicit rules now govern what never appears in user-facing output: no skill names, no internal abbreviations, no phase announcements. Principle #10 (output discipline) and the "framework writes, solo responds" principle were added and propagated across all files. The contract is now the authoritative reference for output behavior across every mode.
Improved
Plain language audit — complete pass across all files
Every file was reviewed and scrubbed of internal abbreviations (SBF), skill names in user-facing text, and technical shorthand. The framework's external voice now matches the output contract fully.
New
Named project resume
Saying "auto pilot on [project name]" now picks up any in-flight project without needing a saved prompt or prior setup. Context is restored automatically at session start.
New
Nivya — on-demand framework and project companion
New skill that recalls both framework mechanics and active project context on demand. Distinct from framework-health (always-on health monitor) and product-continuity (session state tracker). Nivya is invoked when you need to ask a question about how the framework works or where a project stands.
Improved
Design Sprint — pre-flight gate + Figma layer enumeration
A pre-flight gate now runs before any HTML is produced in Design Sprint on-ramp 2. Figma frame enumeration happens upfront — all frames and layers are captured before the sprint begins, not discovered mid-build. Fidelity standard tightened to require extracting exact Figma values (not approximations).
Improved
framework-health — gate confirmations, milestone lines, fork format
Explicit gate confirmations and milestone lines added. Two-path fork format standardized. Anti-patterns for vague transitions documented. Framework-health now checks for upstream updates at mode activation.
April 23
Workshop, Gates, and the Phase Guide Library
New
Workshop framework — separate path for spike work
Spike and tool-shaped work now has its own entry, its own skills (scope-check, spike, land), and a clear boundary with the main build flow. Workshop work doesn't go through the phase chain — it enters, explores, and lands as a defined artifact or a decision to proceed.
New
Onboard skill — bring existing projects into the framework
Projects that predate the framework can now enter it without repeating discovery from scratch. Onboard maps prior work to framework anchors and picks up where the project actually is.
New
Phase gates throughout — explicit confirmations at every phase end
Every phase now ends with a named gate: actual output files listed, followed by a direct two-option prompt — start the next phase now, or close out here. Passive "good stopping point" language retired.
Improved
Process promoted to fourth anchor
The four required anchors are now: design, data, done, and process. No slice reaches Ready without all four. Process is no longer an implicit input — it's a named requirement.
New
Phase Guide Library — nine guides, one per phase
Full walkthroughs added for every phase: Brainstorm, Discover & Tech Context, Design Sprint, Data & Design Review, Plan, Build, QA & Acceptance, Phase Test, and Deploy. Each guide covers what to expect, what gets produced, and what the gate looks like.
New
Getting Started + FAQ
Two new entry-level docs. Getting Started covers first sessions, entry points, session modes, and session hygiene. FAQ answers real questions from mid-discovery, mid-design, mid-build, QA, and out-of-order situations.
Improved
design-review — Tier 2 multi-agent orchestrator + stakeholder handoff
Tier 2 multi-agent review added for high-stakes design decisions. Stakeholder handoff package now produced at the end of both design-sprint and design-review — not just on request.
New
Session hygiene formalized
Close loop, resume prompt, and model guidance are now a defined framework pattern — not ad hoc. Sessions have a consistent shape: open with context restore, close with a handoff artifact.
Improved
Language audit — "visual contract" and "decision surface"
"Rosetta Stone" retired as a metaphor across all files. Replaced with "visual contract" (the design artifact as a binding reference) and "decision surface" (what the design review exposes for a decision). More precise, less metaphor-dependent.
April 22
Governance Layer and Communications Launch
New
Framework Curator — governance for framework changes
Any change to the framework now has a defined path: read the affected files, draft a cascade proposal, preview every change, wait for approval, execute, then validate integrity. Drift can no longer accumulate silently — every change is deliberate and documented.
New
Communications hub launched
Index page, landing structure, and robots.txt added — the comms docs now have a proper home rather than a flat directory of files.
Improved
Reference material separated from framework skills
Reference docs moved out of the skills directory. Clear boundary between what the framework does (skills) and what it documents (reference). Skills stay lean; reference grows without cluttering the skill set.
Improved
product-continuity — phase marker added
current-phase.md now tracks the active phase across sessions. Context restore at session start picks this up automatically.
April 21
Framework Ships
New
Solo Builder Framework — complete skill set
The full framework shipped: all phase skills, always-on skills, support skills, and the complete phase chain from Brainstorm through Deploy. The foundation everything else is built on.
New
Deploy and Retrospective skills
Closed the back end of the phase chain. Deploy handles the production launch sequence. Retrospective captures what was learned at the end of every build cycle so it feeds the next one.
New
Process anchor — integrated from day one
Process sketch added as a moment in brainstorming. Process anchor wired into prd-to-plan. Process became a first-class input from the very start — not retrofitted later.
Improved
process-mapper integrated into design-review
The to-be process map now feeds directly into design review. Design decisions are grounded in the process they serve, not evaluated in isolation.
New
Communications suite — shipped alongside the framework
Process map, Executive Deck, Solo Builder Deck, and Skills Reference all launched on day one. The framework shipped with a complete external story, not just internal tooling.
Improved
Portable setup
README and Cursor user rules updated so the framework works on any machine without local path assumptions. Fresh-machine install works without manual fixes.
