Solo Builder Framework

Release Notes

What changed in the framework, what was added, and why — in plain language.

May 1
Built to Ship, Not Just to Pass
New
Quality contract — four required categories on every slice
Every slice now requires a quality contract written during design review, before build starts. The contract is structured into four required categories: Failure states (what happens when an external call fails or times out), Edge cases (empty data, max values, concurrent actions), Input validation (what is rejected, when, with what feedback), and Security (user input sanitized, data scoped, auth server-side). Each category must be addressed or explicitly marked N/A with a reason — a blank category is not valid. This is not a suggestion layer on top of done criteria. It is the pre-written spec the builder is held to, and the contract the code review checks against line by line. A slice without a quality contract cannot reach Ready.
New
Code review Check 8 — adversarial quality contract verification
Code review now reads the quality contract before reading the code, then checks every contract line against the implementation with deliberate adversarial specificity. A try/catch that swallows the error silently does not satisfy "user sees an error message." A validation check on the wrong event does not satisfy "rejects empty string at submission." The contract is the spec. The code either satisfies it or it doesn't — there is no partial credit and no nearest-thing rationalization. Check 8 failures route back to build, same as the seven structural checks that preceded it. This is the mechanism that closes the gap between "it works in testing" and "it holds under real conditions."
New
Code review Check 9 — independent security baseline
A fixed five-item security gate now runs on every slice, independent of what the quality contract says. The five sub-checks: input sanitization (no raw rendering of user-supplied content), data scoping (data returned scoped to the authenticated user), server-side auth (authorization logic lives on the server — a hidden UI element is not access control), injection prevention (user input not string-concatenated into queries or commands), and no secrets in client-visible code. Check 9 is independent by design: a quality contract might mark security N/A for a slice with no user input or external calls, but Check 9 still runs. It catches classes of vulnerability a per-slice contract writer might not surface. Any sub-check failure routes back to build.
Improved
Self-verification now walks observable contract items
Before declaring code-complete, the builder is now required to walk every observable quality contract item in the running app — not just the happy path. This means: simulating a failure and confirming the error state renders, submitting empty or invalid input and confirming validation fires at the right point, and for any security contract line, confirming user-supplied content is not rendered raw. The build plan also now includes explicit steps for these observations — so the builder is implementing contract requirements as discrete steps, not hoping they emerge from the feature work.
April 30
Build Discipline, Session Hygiene, and Observability
Improved
Pre-build read gate — verbatim quote required
The pre-build read gate in solo-build has been tightened: the builder must now produce a verbatim quote from the design file — the exact string, word for word — before writing any code. A paraphrase no longer satisfies the gate. This closes the loophole where a builder could confirm the read from memory or a prior session rather than actually opening the file. The distinction matters: a design file that has changed since the last session, or one the builder recalls slightly wrong, produces implementation drift that compounds across slices. The verbatim quote makes the read verifiable.
New
Session signal system — passive telemetry across all machines
Four skills now append one-line signals to a temporary file when notable failure events occur during a build session: stuck protocol fired, slice rebuilt, priority deviation (solo-build); QA caught a missed self-verification (solo-qa); code review failed (code-review-and-quality); phase test on hold (phase-test). A Stop hook wired via onboard reads the file at session end, appends structured lines to a shared session log in the engineering-playbook repo, pushes, and clears the file. If push fails, signals are held in a pending file and included in the next successful push. The log accumulates across all machines and projects — a signal-level view of where build quality issues surface most.
New
Handoff staleness hook — automatic warning when backlog outpaces handoff
Onboard now wires a Stop hook into every project: at the end of each session, if the backlog has been modified more recently than the handoff document, a warning surfaces. The handoff is what the framework reads at session start to orient — if the backlog changed and the handoff didn't, the next session opens on stale context. The hook is silent when handoff is current. Existing projects can add it manually via the command in the onboard skill; new projects get it automatically.
New
Observability tool decision locked at tech-context
Tech context now explicitly asks whether runtime observability is required and which tool is in place — Datadog, CloudWatch, or none. The choice is mutually exclusive and locked at tech-context time. If a tool is declared, an observability setup infrastructure slice is required in the backlog before any feature slices start. This prevents the common pattern of shipping features without instrumentation and wiring it in after the fact. For projects with an established org standard, that standard is declared here — not inferred from prior context or assumed by the builder.
April 30
Continuity, Collaboration, and the Cycle After Launch
New
Onboard — companion compatibility pass
Onboarding an existing project now ends with a four-check compatibility pass that verifies the project will sync cleanly with the Solo Companion before exiting. The checks: project registered in the framework registry, backlog has the three required section headers, handoff contains an Open right now section, and every slice record has a Review URL field. A project that silently fails any of these shows up in the companion with empty dashboards and missing cards. The pass fills or flags each gap — so the first companion sync after onboarding just works.
New
Shared ideas backlog — framework collaboration with owner sign-off
The framework curator now supports a shared ideas backlog at shared/ideas.md. Any contributor can brainstorm with the curator and log proposed ideas — the curator handles the write, commit, and push. Ideas sit as Proposed until the framework owner reviews them. At the start of every owner session, the curator surfaces pending items for sign-off. Approved ideas can then move through the normal curator workflow to become real framework changes. Nothing proceeds to build without owner approval. Contributor installs redirect direct framework file edits to the ideas backlog automatically — preventing accidental changes outside the curator workflow.
New
Deployed project — new cycle routing in start
When a project is opened and all its phases are at Deployed status, the framework no longer tries to continue the old work. It detects the deployed state, orients on what was shipped, asks what new work is being added, and opens a new phase in the existing backlog. Completed phases stay exactly as they are — read-only history. The new phase inherits the full context of the project (decisions, process map, tech context, design artifacts) without requiring any of it to be recreated. The handoff updates to reflect the new cycle. One door in — the same entry point handles new projects, continuing projects, and returning to shipped ones.
April 27
Canonical Records — The Backlog Gets a Specification
New
Records specification — canonical format for all three record types
The backlog now has a single authoritative specification for what every record must contain. Every skill that reads or writes backlog records reads this document first. It defines the full field set for Slice (14 fields), Deliverable (13 fields), and Phase (14 fields) — including status structures, field rules, the skill ownership table, re-phasing protocol, and the decisions and change log format. No skill invents fields. No skill omits required fields. The spec is the contract.
New
Two descriptions required on every record — plain language and technical
Every slice, deliverable, and phase now carries two independent descriptions. The plain language description is written for the solo — what they will see, have, or be able to do when the work is complete. The technical description is written for the builder — implementation approach, constraints, edge cases. Neither copies from the other. Neither summarizes the level below it. A phase description is not a summary of its deliverables. A deliverable description is not a summary of its slices. Writing them fresh at every level forces the author to think about what each object actually delivers, independently.
New
Three-level verification — self-verification checklist, acceptance criteria, builder confirmation
Every record type now carries the same three verification fields. The self-verification checklist is what the builder runs before presenting to the solo. The acceptance criteria is the shared contract — the same list the builder checks and the solo reviews, independently. The builder confirmation is populated at presentation time with ✓/✗ results and observations, so the solo arrives at review with the builder's work already shown. Each level checks what only that level can see: slices verify individual elements, deliverables verify slices working together, phases verify deliverables working end-to-end.
New
In Test — new slice status between solo QA sign-off and Done
Slices now move to In Test after solo-qa sign-off, not directly to Done. Done is reserved for phase-test completion — when the full phase has been verified end-to-end with real data. This resolves a genuine ambiguity: a slice that passed solo QA against mock data and a slice that has been confirmed working in the complete, integrated phase are meaningfully different things. In Test makes that visible in the backlog. The full chain: In Build → In QA → In Test → Done. Phase-test is what closes the final gap.
New
Integration — third deliverable type
Deliverables now have three types: Screen, Logic, and Integration. Integration connects real data to a Screen deliverable — it may contain both UI slices and data integration slices, always sequences after its Screen companion is Accepted, and can spawn new slices when real data reveals gaps. This makes the mock-to-real transition explicit in the plan rather than buried in individual slices. An Integration deliverable is Accepted only when every gap revealed by live data has been addressed — including any newly discovered slices.
New
Re-phasing protocol — formal process for moving work between phases
When a slice or deliverable needs to move to a different phase mid-build, a 7-step protocol now executes in full: solo confirms the move → decision logged → moved item updated → source phase "Explicitly out of scope" updated → destination phase deliverables list updated → source phase re-assessed (does it still answer its question?) → destination phase re-assessed (does the moved item fit?). All seven steps happen in the same action. The decision log entry and the record updates are never separated. No quiet edits.
New
Backlog Status Reference — standalone reference page
New page in the comms docs showing all three status grids side by side — Slice (8 statuses), Deliverable (4 statuses), Phase (4 statuses) — with flow chains, plain-language meanings, who sets each status, and the skill ownership table. Linked from the index under Framework Overview. When the backlog shows an unfamiliar status, this is the one-click answer.
Improved
Build, QA, and phase test guides aligned with the new record format
The build guide, QA guide, phase test guide, and skills reference all updated to reflect the new status chain and record format. Builder self-verification step added to the code-complete protocol. In Test introduced throughout the QA flow. Done definition table updated to show phase-test as the final step. Phase test readiness and handoff sections updated to match. Six skill cards in the skills reference updated for the new status names and deliverable fields.
April 25
Delivery Rules — Commands and Review
New
AI runs terminal commands — never the solo
The framework never pastes a command for the solo to copy and run. If a command needs to execute, the AI runs it. If the action is significant or destructive, the AI asks permission first — then runs it. Applied as a load-bearing rule in the Output Contract, framework principles, solo-build skill, and both installed Cursor rules.
New
Review is always visual
When a build produces UI or HTML output and review is needed, the AI serves it as a viewable page before asking for feedback. The solo looks at the work, then responds. Asking for sign-off on a description of what was built is not review. Same six-file cascade as the terminal commands rule.
New
Release notes blog launched
This page. Framework improvements are now tracked in plain language — what changed, what was added, and why — rather than living only in git commit messages.
New
Deliverable — new framework concept
The framework now has a named layer between the plan and individual slices. A deliverable is the body of work a set of slices collectively produce. Two types: screen (reviewed visually in the browser) and logic (reviewed against evidence — test results, data state, or a simulated flow through affected screens). Three fields per deliverable: technical spec for the AI, solo description for the human, and acceptance criteria as the shared contract both parties build and review against. Documented across the framework doc, QA guide, Phase Test guide, skills reference, and getting-started.
New
Two review levels
Slice approval (did this unit of work complete correctly) and deliverable acceptance (is the full body of work done and does it match the agreed contract) are now distinct moments. The AI self-verifies against acceptance criteria before presenting a review to the solo. QA runs a deliverable-level check once all slices in a deliverable reach Done — screen deliverables get a visual pass, logic deliverables get an evidence pass. Phase Test walks each deliverable in the phase and confirms they hold together in integration.
New
"show progress" and "show plan"
Two plain-language phrases available at any time during build. "show progress" returns the active deliverable — all slices, current status, acceptance criteria. "show plan" returns the full build — all deliverables, delivered / in progress / backlog. Surfaced automatically at key moments; also available on demand. Noted in the getting-started Build phase row so solos know the phrases from day one.
Improved
Session modes renamed: bare / piloted / guided
The three session modes were renamed from bare / assisted / auto-pilot. "Auto-pilot" created the wrong mental model — people assumed the AI worked independently while they stepped away. The new names reflect what each mode actually means: bare (you invoke, nothing pre-loads), piloted (you're at the controls, always-on instruments are live), guided (the framework leads the phase sequencing, you're in every decision). Cascaded across 18 files — framework doc, all always-on skill files, template, setup prompts, and every communications page.
New
Dedicated modes slide in the solo deck
New Slide 5 explains bare / piloted / guided with one paragraph per mode — clear enough that the mental model lands before the always-on slide references them. Includes an enhanced mode callout at the bottom explaining it as a separate capability layer (Claude Code only, multi-agent orchestration) that can combine with any session mode.
New
Data integration planning — data questions log + integration deliverables
The mock→real data swap is no longer treated as a checklist item at phase test. It's now explicitly planned from design review onward. Design review gets a named data behavior pass — required for every screen with external data — that asks scale and behavior questions upfront: will this list paginate, which fields can be null, what does the empty state look like? Every unanswered question goes into a data questions log that gates dependent UI slices from reaching Ready. Mock data is updated after questions resolve, so UI slices are built against realistic conditions from the start. In planning, every Screen deliverable with external data gets a companion Logic deliverable for data integration — always a separate deliverable, always sequencing after the Screen is accepted, scoped only once the data questions log is resolved. Standard done criteria for integration slices cover API shape, data-mapping.md confirmed, pagination and filtering, empty and error states, loading states, and auth. At integration deliverable acceptance, solo-qa runs a UX impact check: does live data produce anything a user would see that the design didn't account for? If yes, the affected UI slices reopen and design review runs before the deliverable is accepted. Early questions, separate deliverables, late confirmation.
New
process-change — to-be map change management
New skill providing a consistent protocol for changes to the agreed to-be process map, regardless of cause or where in the build they surface. Stakeholder input, demo gaps, technical constraints, and solo realizations all receive the same seven-step response: capture trigger → document change → three-tier impact scan (scheduled, in-progress, and completed work — nothing assumed safe) → solo decides → execute → re-prioritize → solo confirms before resuming. Wired into solo-build (map-level conflict auto-detection), qa-triage (new map-level gap classification), and phase-test Stage 6 (process-level mismatch blocks the gate). The process is the contract. When it changes, it changes deliberately.
Improved
Phase test timing calibration — readiness table + framework-health nudge
The "When to Invoke" judgment call now has structure. A three-state readiness table (Ready / Not Yet / Overdue) gives new solos the calibration guidance the gate criteria alone couldn't provide. The Overdue state is new — it names the scope-creep failure mode where a solo keeps adding slices rather than pulling the gate trigger on a finished phase. framework-health gets a matching signal: when all current-phase slices hit Done with no open build qa-triage items, it fires a single nudge before stepping back. Hole #6 resolved.
Improved
Long-gap recovery — framework-health detection + product-continuity re-entry protocol
When a session resumes after 21 or more days, the framework now detects the gap and responds to it. framework-health parses the handoff date at mode activation — the same read it was already doing — and if the gap is 21+ days, surfaces a re-entry prompt before any work begins. product-continuity handles the re-entry as three sequential checks: codebase state (git log since last session, uncommitted changes, open branches), environment (does the app run, any dependency drift), and context validity (blocking questions that may have been answered, assumptions that may have gone stale, key context that may have shifted). A re-entry summary closes the check and hands back to normal session flow. Short gaps are silent. No new skill — detection wired into an existing signal, re-entry wired into existing context documents.
April 24
Output Discipline & Design Sprint Precision
New
Output Contract — two new load-bearing principles
Explicit rules now govern what never appears in user-facing output: no skill names, no internal abbreviations, no phase announcements. Principle #10 (output discipline) and the "framework writes, solo responds" principle were added and propagated across all files. The contract is now the authoritative reference for output behavior across every mode.
Improved
Plain language audit — complete pass across all files
Every file was reviewed and scrubbed of internal abbreviations (SBF), skill names in user-facing text, and technical shorthand. The framework's external voice now matches the output contract fully.
New
Named project resume
Saying "auto pilot on [project name]" now picks up any in-flight project without needing a saved prompt or prior setup. Context is restored automatically at session start.
New
Nivya — on-demand framework and project companion
New skill that recalls both framework mechanics and active project context on demand. Distinct from framework-health (always-on health monitor) and product-continuity (session state tracker). Nivya is invoked when you need to ask a question about how the framework works or where a project stands.
Improved
Design Sprint — pre-flight gate + Figma layer enumeration
A pre-flight gate now runs before any HTML is produced in Design Sprint on-ramp 2. Figma frame enumeration happens upfront — all frames and layers are captured before the sprint begins, not discovered mid-build. Fidelity standard tightened to require extracting exact Figma values (not approximations).
Improved
framework-health — gate confirmations, milestone lines, fork format
Explicit gate confirmations and milestone lines added. Two-path fork format standardized. Anti-patterns for vague transitions documented. Framework-health now checks for upstream updates at mode activation.
April 23
Workshop, Gates, and the Phase Guide Library
New
Workshop framework — separate path for spike work
Spike and tool-shaped work now has its own entry, its own skills (scope-check, spike, land), and a clear boundary with the main build flow. Workshop work doesn't go through the phase chain — it enters, explores, and lands as a defined artifact or a decision to proceed.
New
Onboard skill — bring existing projects into the framework
Projects that predate the framework can now enter it without repeating discovery from scratch. Onboard maps prior work to framework anchors and picks up where the project actually is.
New
Phase gates throughout — explicit confirmations at every phase end
Every phase now ends with a named gate: actual output files listed, followed by a direct two-option prompt — start the next phase now, or close out here. Passive "good stopping point" language retired.
Improved
Process promoted to fourth anchor
The four required anchors are now: design, data, done, and process. No slice reaches Ready without all four. Process is no longer an implicit input — it's a named requirement.
New
Phase Guide Library — nine guides, one per phase
Full walkthroughs added for every phase: Brainstorm, Discover & Tech Context, Design Sprint, Data & Design Review, Plan, Build, QA & Acceptance, Phase Test, and Deploy. Each guide covers what to expect, what gets produced, and what the gate looks like.
New
Getting Started + FAQ
Two new entry-level docs. Getting Started covers first sessions, entry points, session modes, and session hygiene. FAQ answers real questions from mid-discovery, mid-design, mid-build, QA, and out-of-order situations.
Improved
design-review — Tier 2 multi-agent orchestrator + stakeholder handoff
Tier 2 multi-agent review added for high-stakes design decisions. Stakeholder handoff package now produced at the end of both design-sprint and design-review — not just on request.
New
Session hygiene formalized
Close loop, resume prompt, and model guidance are now a defined framework pattern — not ad hoc. Sessions have a consistent shape: open with context restore, close with a handoff artifact.
Improved
Language audit — "visual contract" and "decision surface"
"Rosetta Stone" retired as a metaphor across all files. Replaced with "visual contract" (the design artifact as a binding reference) and "decision surface" (what the design review exposes for a decision). More precise, less metaphor-dependent.
April 22
Governance Layer and Communications Launch
New
Framework Curator — governance for framework changes
Any change to the framework now has a defined path: read the affected files, draft a cascade proposal, preview every change, wait for approval, execute, then validate integrity. Drift can no longer accumulate silently — every change is deliberate and documented.
New
Communications hub launched
Index page, landing structure, and robots.txt added — the comms docs now have a proper home rather than a flat directory of files.
Improved
Reference material separated from framework skills
Reference docs moved out of the skills directory. Clear boundary between what the framework does (skills) and what it documents (reference). Skills stay lean; reference grows without cluttering the skill set.
Improved
product-continuity — phase marker added
current-phase.md now tracks the active phase across sessions. Context restore at session start picks this up automatically.
